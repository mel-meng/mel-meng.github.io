Key conclusion

The data is organized as a reference model: the node stores the DWF assignment and selected pattern name; the temporal variation values live in a reusable global database item. The link is a string reference from DATA T_DWF_TVAR "8-2-24" ... "Daily1" to GLDBITEM "Temporal Variations" "Daily1" and matching GLDBDATA rows.

Tool-layer implication: do not flatten the temporal variation into every node as if it were node-owned data. Represent it as a reusable object and make the node point to it.

Data model

Three questions for the exchange-data spec

1. What is tied to a node?

The selected DWF method, units, base values, enable flags, and the reference name of the temporal variation. In this sample, those are DATA rows whose object key is "8-2-24".

2. What is tied to a reusable database item?

The named temporal-variation item and its multiplier arrays. In this sample, Daily1 is a GLDBITEM under "Temporal Variations", with T_HV and T_DV stored as GLDBDATA.

3. How are they linked?

By name, scoped by library category. The node field T_DWF_TVAR = "Daily1" points to the global database item ("Temporal Variations", "Daily1").

Linkage contract

Observed link

DATA T_DWF_TVAR "8-2-24" 0 1 "Daily1"
           │
           └── matches item name

GLDBITEM "Temporal Variations" "Daily1"
GLDBDATA T_HV "Temporal Variations" "Daily1" 24 ...
GLDBDATA T_DV "Temporal Variations" "Daily1" 7 ...

Inferred high The match is name-based. The safe key for the library item is the pair (library_name, item_name), not item_name alone.

Exchange validation rules

  1. For each node DWF assignment, read T_DWF_TVAR as a reference, not as the multiplier array.
  2. Resolve the reference against GLDBITEM "Temporal Variations" <name>.
  3. Require or report T_HV count 24 and T_DV count 7 for a complete DWF pattern.
  4. Preserve unused temporal variations unless the export tool explicitly prunes library items.
  5. When writing XPX, create/update the reusable item once, then point nodes to it by name.

Subcatchment rows use the same predicate as the tool explorer filter hasDwfLoading:

  • sewerInputs.enabled not explicitly false
  • dryWeatherFlow.enabled not explicitly false

Tool contract

Draft specification language for the tool layer

DWF in XPX should be modeled as a node assignment referencing a reusable temporal variation. A node may carry DWF scalar parameters and flags in DATA records keyed by node name. A temporal variation is a named global database item under the Temporal Variations library. The node links to the reusable item using the selected temporal variation name stored in T_DWF_TVAR. A complete temporal variation contains a 24-value hourly multiplier array and a 7-value daily multiplier array.

Non-goals / open items

  • Do not assert complete enum mappings from this single sample.
  • Do not assume all DATA rows with empty object name are irrelevant; they may be model/global settings. For DWF extraction, only selected fields are needed.
  • Do not treat XP Tables as the source of model inputs; it is table/view metadata unless further tests prove otherwise.
  • Do not assume pattern-name uniqueness across all libraries; use library_name + item_name.
